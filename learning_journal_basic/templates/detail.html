  • Concurrency:

    • Synchronous operations cause problems when scaling.
    • Select:
      • Simple concurrency with select built-in can accommodate this:
        • The module provides select, an interface to the underlying Unix select system call. The purpose of select is to take a list of possible I/O channels (file handles, Unix sockets, network sockets) and return lists of those that are ready to be read, written or in an error state.
    • Threading:
    • DON’T USE THE THREAD MODULE DIRECTLY
      • lightweight independent process
      • single CPU
      • share memory with each-other and their parent process
      • convenient but complex
        • Race conditions
        • deadlocks
        • hard to debug
      • You can only ever access one thread at a time.
    • Multiprocessing:
      • Basically same API as threading.
      • Does not share memory.
      • Messages must be passed if shared state is required
      • No Shared Resources == No deadlocks or race conditions.
      • Better choice for Python concurrency.
    • ASync Concurrency:
      • Uses events to handle I/O outside the flow of the main process.
        • Events:
          • e.g. Client socket connecting
      • There are two models for this type of event-driven operation. These two methods are nicely modeled by two well-known packages in the Python ecosystem:
        • Callbacks : twisted package
        • Coroutines. : gevent package
